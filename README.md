# VibecodingTemplate

`VibecodingTemplate` 是一个用于快速启动新项目的 vibecoding 模板仓库。

核心目标是让新项目一开始就具备清晰的 Agent 规则、设计记录、环境固化、调研记录、修复记录和测试验收习惯，避免项目推进到一半后才补流程。

本模板采用 MIT License。使用本模板创建的新项目可以自行选择许可证。

## 目录说明

- `template/`：新项目启动时要复制的模板内容。
- `template/AGENTS.md`：复制到新项目后的 Agent 工作规则，是模板最核心的文件。
- `template/design/`：总体设计、阶段设计、阶段报告和架构决策。
- `template/research/`：调研记录、事实确认、依赖笔记和实验结论。
- `template/fix/`：问题修复记录。
- `template/memory/`：项目内可复用经验和踩坑记录。
- `template/doc/`：面向使用者或开发者的说明文档。
- `template/requirements/`：运行或安装需要的依赖文件和说明。
- `template/references/`：第三方 demo、协议样例、外部文档和工具包。
- `template/test/`：测试入口、测试脚本、测试素材和测试报告。

## 操作流程

### 1. 准备模板

拉取本项目后，把 `template/` 作为新项目的起点。

可以直接在 `template/` 目录中开始，也可以把 `template/` 复制到新的项目目录后再开始。

### 2. 启动 Agent 初始化

在你的工具中启动 Agent，例如 Codex、Claude Code 或 GitHub Copilot。

直接要求 Agent 阅读模板路径里的 `AGENTS.md`，并按其中规则初始化项目。

### 3. 明确项目边界

和 Agent 一起确认这些内容：

- 项目目标
- 目标用户
- 核心使用场景
- 交付边界
- 暂不实现的内容
- 后续开发、记录、依赖和测试规则

确认后，让 Agent 创建或更新 `design/design_doc.md`，把上述内容固化下来。

推荐提示词：

```text
请先不要写代码。请根据 AGENTS.md 的规则，和我一起完成项目初始化的“项目边界确认”。

请用问题清单的方式依次确认：项目目标、目标用户、核心使用场景、交付边界、暂不实现的内容、主要技术约束、后续开发记录方式、测试验收方式。

如果我已经提供的信息不完整，请主动追问；如果信息足够，请先给我一版结构化总结让我确认。确认后，再创建或更新 design/design_doc.md，把项目边界和初始开发约定写进去。
```

### 4. 固化运行环境

在开始安装依赖或运行项目之前，先确定运行环境。

常见选择包括 conda、venv、系统解释器、容器环境或 Node 版本管理方案。

确定后，让 Agent 在 `design/design_doc.md` 中记录：

- 环境名称
- 创建方式
- 激活命令
- Python、Node 等关键运行时版本
- 依赖安装入口
- 依赖记录文件或目录

缺少依赖时，Agent 可以直接安装到已固化的项目环境中；如果涉及版本覆盖、升级、降级、卸载或锁文件大规模改写，需要先征得用户同意。

推荐提示词：

```text
请先不要安装依赖。请根据 AGENTS.md 的规则，和我一起确定本项目的运行环境方案。

请先检查当前项目已有文件和可用环境，再给出 conda、venv、系统解释器、容器环境或 Node 版本管理方案中的推荐选择，并说明理由、创建方式、激活命令、关键运行时版本、依赖安装入口和依赖记录文件。

在我确认环境方案后，再把环境名称、创建方式、激活命令、关键版本和依赖管理规则写入 design/design_doc.md。后续缺少依赖时可以安装到这个已固化环境内；如果涉及版本覆盖、升级、降级、卸载或锁文件大规模改写，必须先询问我。
```

### 5. 处理前端约束

如果项目涉及前端，先创建或确认根目录 `DESIGN.md`。

如果你不知道什么是DESIGN.md，最好的办法是直接去找一个大厂的`DESIGN.md`复制到template/。

如果没有 `DESIGN.md`也没事，模板规则会禁止 Agent 手写原生 CSS，只允许使用成熟外部 UI、样式和图标库，例如 `shadcn/ui`、`Tailwind CSS` 和 `Lucide`。

这些库本身就自带现代审美规范，AI 只要往里套组件，出来的效果也绝对比它自己瞎编强一百倍。

### 6. 记录调研和决策

如果有不清楚的技术点，直接让 Agent 调研。

涉及第三方 API、协议、依赖、模型、工具或技术方案时，Agent 会自动把事实依据和结论写入 `research/`并自行查阅。

### 7. 进入开发循环

初始化完成后，再进入具体功能实现。

后续按这个循环推进：

1. 明确本轮任务。
2. 必要时更新设计或调研记录。
3. 实现功能或修复问题。
4. 运行与改动范围匹配的验证。
5. 把修复记录、阶段报告、测试结果或可复用经验写入对应目录。

具体目录和记录要求以 `template/AGENTS.md` 为准。
