# AGENTS.md

## 项目目标

- 本项目由 `VibecodingTemplate` 启动，目标是在明确需求后快速进入设计、实现、验证和复盘闭环。
- 正式开发前，把本节改成当前项目的具体目标、目标用户、关键场景和交付边界。
- 如果项目目标尚不清晰，先通过 `design/design_doc.md` 固化已知信息、待确认问题和第一阶段范围，再开始大规模实现。

## 工作原则

- 优先完成用户当前明确要求；当用户已经要求实现、修复或生成内容时，可以直接编辑相关文件。
- 遇到需求冲突、破坏性操作、不可逆数据变更、权限/费用/安全风险或多种方案影响明显不同的情况，先暂停并向用户确认。
- 模块化开发，避免巨体量单文件；新增抽象必须服务真实复杂度，不为了形式拆分。
- 尊重现有代码风格、目录结构和技术栈；在没有必要时不引入新框架、新构建链或重型依赖。
- 不写入密钥、令牌、账号密码、私有证书、本机绝对路径和只属于个人环境的配置。
- 项目文档默认使用中文；API 路径、协议字段、状态枚举、命令、代码标识、文件名、模型名和原始报错保留英文原文。

## 新项目启动流程

- 创建或更新 `design/design_doc.md`，写清项目目标、用户场景、核心功能、非目标、技术选型、目录规划和阶段拆分。
- 项目启动时确定运行环境方案，例如 conda 环境、venv 环境、系统解释器或容器环境；确定后写入 `design/design_doc.md`，并记录环境名称、创建方式、激活命令、Python/Node 等关键运行时版本和依赖安装入口。
- 如果已有阶段计划，阶段详细设计写进 `design/stage*_doc.md`；开始某阶段开发前先阅读对应阶段文档。
- 需要事实依据、第三方 API、协议细节、性能数据或依赖选择时，先在 `research/` 记录调研结论。
- 重要架构决策、模块边界和长期约定写回 `design/design_doc.md`，避免只停留在对话里。
- 完成阶段后，在 `design/stage*_report.md` 记录完成范围、验收结果、关键测试、遗留问题和下一阶段建议。

## 目录约定

- `design/`：总体设计、阶段设计、阶段报告和架构决策。
- `research/`：调研记录、事实确认、依赖笔记、实验结论和 example 验证总结。
- `fix/`：小规模 bugfix、运行时问题修复和阶段验收阻塞修复记录，文件名建议为 `<date>-<topic>.md`。
- `memory/`：项目内可复用经验、重复失败的处理方式和环境相关教训。
- `doc/`：面向使用者或开发者的说明文档。
- `requirements/`：运行或安装需要的依赖文件，例如 SDK 安装包、运行时二进制、Python 扩展或系统依赖说明。
- `references/`：第三方 demo、协议样例、外部文档和工具包；业务代码不要写进该目录。
- `test/<test_name>/`：测试入口、测试脚本、小型固定素材和人工整理的 `report.md`。
- `test/<test_name>/output/`：测试运行输出、日志、截图、缓存、构建产物和临时文件；默认不进入 git。

## 拼写和引用约定

- `references/` 是唯一正确的参考资料目录名。
- `refrences/` 是常见拼写错误；遇到用户或文档写成 `refrences/` 时，默认按 `references/` 理解，不要新建或迁移到错误拼写目录。
- 如果用户没有明确要求参考 `references/` 中的代码，不主动查阅参考目录里的代码。
- 新增或更新第三方参考代码库时放在 `references/`；新增或更新运行依赖时放在 `requirements/`。

## 设计和记录

- 大型功能、阶段方案和架构决策优先进入 `design/`。
- 小规模修复记录到 `fix/<date>-<topic>.md`；简单修复可以简写，但要保留“现象-根因-验证”闭环。
- `fix/` 记录建议包含问题背景、现象、根因、修复计划、修改范围、验证结果、遗留风险和关联记录。
- 完成有价值的小型 example 验证后，在 `research/` 生成对应研究记录，说明验证目标、命令、结论、关键结果和对设计/开发的影响。
- 遇到非显然失败、重复失败、环境/依赖/硬件相关失败时，先查看 `memory/` 是否有可复用教训。
- 解决值得复用的失败后，把失败现象、原因和处理办法补充进 `memory/`。

## 运行和依赖

- 优先使用项目已固化的包管理器、虚拟环境、脚本入口和锁文件；开始运行、测试或安装依赖前，先确认当前 shell 已进入项目约定环境。
- 如果项目尚未固化运行环境，先和用户确认使用 conda、venv、系统解释器、容器或其他方案，再创建/选择环境并记录到 `design/design_doc.md`。
- 运行环境确定后，缺少依赖时可以直接在该环境内安装并继续推进，同时同步更新对应依赖记录；不要安装到未确认的全局环境。
- 如项目使用 Python，则新增 Python 依赖安装到已固化的 Python 环境；依赖清单、安装包和运行时二进制仍按 `requirements/` 目录约定管理。
- 涉及依赖版本覆盖、升级、降级、卸载、锁文件大规模改写或可能影响现有功能的依赖解析变更时，先向用户说明影响并获得同意。
- 非 Python 依赖按风险、体积、权限和系统影响判断是否需要先确认；系统级依赖、驱动、服务、数据库或外部工具安装默认先确认。
- 不把依赖安装目录、缓存目录、构建产物或本机临时文件提交到仓库。

## 前端开发约束

- 只要任务涉及前端页面、组件、样式、布局、交互、动效或视觉调整，开始实现前先阅读项目根目录的 `DESIGN.md`。
- 如果不存在 `DESIGN.md`，禁止手写任何原生 CSS；只能使用成熟的外部 UI、样式和图标库完成界面，例如 `shadcn/ui`、`Tailwind CSS` 和 `Lucide`。
- 在没有 `DESIGN.md` 的情况下，禁止新增或修改手写 `.css`、`.scss`、`<style>`、CSS Modules、CSS-in-JS 和复杂内联 `style`；只允许必要的库配置、库生成代码和基于库约定的类名组合。
- 如果 `DESIGN.md` 已存在，优先遵循其中的设计系统、组件规范、色彩、间距、响应式和动效约定；若其内容与用户当前要求冲突，先向用户确认。

## 测试和验证

- 修改代码、脚本、配置或交互后，运行与改动范围匹配的最小有效验证。
- 修复 bug 时验证应覆盖复现路径；新增功能时验证应覆盖主要成功路径和至少一个关键边界。
- 修改前端代码、样式、配置或前端交互后，必须做浏览器检查，记录页面地址、浏览器/视口、交互路径、控制台错误、截图和结论。
- 前端截图必须经过视觉检查，确认是否存在元素遮挡、布局溢出、文字难读、按钮不可点或交互不方便的问题。
- 测试、联调或前端检查完成后，退出本轮测试涉及的项目进程，包括前端 dev server、后端 runtime、浏览器自动化、临时脚本和测试命令；除非用户明确要求保留运行态。

## 交付习惯

- 交付时简要说明修改了什么、验证了什么、还有哪些已知风险或未完成项。
- 如果没有运行测试，明确说明原因。
- 不用长篇复述实现过程；优先给出对用户继续开发有用的信息。
